# DR-2026-03-26-01: System Admin Recurring-Series Management

- **Status:** Accepted
- **Date:** 2026-03-26
- **Deciders:** Ranked Choice maintainers

## Context

System admins can already manage individual votes from `/system`, but recurring vote cleanup still depended on per-vote admin secrets for full recurrence controls. This created an operational gap when recurring votes became orphaned ("rogue") and the vote admin secret was unavailable.

## Options Considered

- **Option A:** Keep recurring controls only in vote admin and require per-vote admin secrets.
- **Option B:** Extend `/system` with recurring-series visibility and management actions protected by `ADMIN_SECRET`.
- **Option C:** Add a one-off script for manual database cleanup outside the app.

## Decision

Adopt Option B.

Add system-admin recurring-series management in `/system` with:
- A full recurring-instance list grouped by `recurrence_group_id`
- A group-level stop action to disable future auto-created votes
- Instance-level delete actions for rogue recurring votes

## Rationale

System admins already own operational responsibility and have `ADMIN_SECRET` access. Keeping recurring cleanup in the same authenticated surface reduces recovery time and avoids brittle out-of-band database interventions.

## Consequences

- **Behavior:** System admins can manage recurring votes globally, even without per-vote admin secrets.
- **UX:** `/system` now has a dedicated recurring-series management section.
- **API:** Adds admin recurrence endpoints for listing recurring instances and stopping recurrence groups.
- **Security:** Actions remain gated by `ADMIN_SECRET` and do not loosen vote-level secret checks for non-admin flows.
