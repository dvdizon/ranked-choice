**Status:** Accepted  
**Date:** 2026-03-25  
**Deciders:** Project maintainers

#### Context
Recurring votes can automatically spawn new vote instances after each cycle. Deleting a single vote instance did not resolve user confusion because recurrence could continue creating replacement votes from the same group. The vote admin panel needed direct recurrence controls without requiring system-admin access.

#### Options Considered
- **Option A:** Keep recurrence management only in system admin.
- **Option B:** Add recurrence controls to vote admin, authenticated by vote admin secret.

#### Decision
Choose **Option B**. Add vote-admin APIs and UI to:
- List vote instances in the same recurrence group
- Delete specific recurring vote instances
- Stop recurrence for the group so no new instances are auto-created

#### Rationale
Vote owners already have the admin secret for their contest and expect to manage recurrence from the same admin page where ballots are managed. This reduces operational friction and resolves repeated vote regeneration issues without broad system privileges.

#### Consequences
- **Behavior:** Vote admins can directly stop recurring schedules and remove unwanted recurring vote instances.
- **UX:** Recurrence state is visible in vote admin with explicit stop/delete actions.
- **Technical constraints:** Adds `/api/votes/:voteId/recurrence` endpoints secured by write-secret verification.
- **Future flexibility:** Maintains separation from system-level ADMIN_SECRET flows while enabling self-service recurrence control.
